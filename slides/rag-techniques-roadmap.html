Mục tiêu cuộc họp

Chúng ta không chỉ “thêm technique”, mà cần biết kỹ thuật nào thật sự mở khóa chất lượng, kỹ thuật nào chỉ trùng với Reducto, và kỹ thuật nào đo được sạch.

offline eval ablation A/B test rollout

Chúng ta đã có gì

  • Ingestion: PDF đi qua Reducto agentic, fallback PyMuPDF math-aware; URL ingestion và text ingestion API đã có.
  • Structure: Reducto trả về content và embed_text, backend normalize rồi lưu page, section, text, embed_text.
  • Index: SQLite local cho documents/chunks/sessions/messages, BM25 cho sparse search, turbovec cho dense search, embedding cache để giảm chi phí.
  • Agent: Router no_retrieval / simple / complex, planner, distill, sufficiency check, answer verify, regenerate.
  • Frontend: Upload PDF/URL, chat, citation hover/click, agent trace/graph, session persistence qua reload.
Eval hiện tại: trên bộ 20 câu, cấu hình Hybrid + Rerank đạt Recall@5 = 0.845, MRR@5 = 0.863, Hit@5 = 1.000. Tức là baseline đã mạnh, nên technique mới phải chứng minh uplift thật.

Những failure mode còn lại

  • Chunk granularity trade-off: chunk nhỏ hơn giúp công thức đúng hơn, nhưng có thể làm recall giảm khi evidence bị phân mảnh.
  • Router edge cases: router là LLM-based nên vẫn có thể misroute một số query mơ hồ, compound, hoặc thiếu context.
  • Multi-hop sensitivity: query cần nhiều bước suy luận vẫn có thể rơi vào replan / sufficiency / early-stop nếu corpus thiếu tín hiệu.
  • Diversity còn mỏng: top-k vẫn có thể gần-duplicate nếu cùng một ý được lặp trong corpus hoặc query quá phổ biến.
Thông điệp cho team: Hãy coi các technique dưới đây như các “lớp điều khiển” cho retrieval, không phải các mẹo tách rời. Đặt đúng thứ tự sẽ quan trọng hơn việc bật thật nhiều thứ cùng lúc.

Nếu dùng Reducto thì nên nghĩ thế nào?

  • Đừng cố “cải thiện chunking” trước khi chứng minh retrieval/query-time còn nghẽn.
  • Ưu tiên technique mà Reducto chưa làm hộ: query rewrite, routing, diversity, context assembly.
  • Với PDF toán, Reducto đang là parse baseline rất mạnh, nên các experiment phải tách biệt rõ.

Rule để không tự đánh lừa mình

  • Nếu một technique chỉ tăng score nhờ làm text dài hơn, chưa chắc là thật sự tốt hơn.
  • So sánh phải giữ cố định parse layer, embedding model, reranker và eval set.
  • Đánh giá riêng retrieval quality, generation quality và latency/cost.
Kết luận thực dụng: Nếu đang dùng Reducto cho PDF, thì các experiment “sạch” nhất là query enhancement, adaptive routing, diversity / hierarchical retrieval, và RSE / compression. Semantic chunking và contextual headers chỉ nên thử khi bạn tách được baseline Reducto.

Query Enhancement

Nhóm kỹ thuật này xử lý câu hỏi trước khi retrieve. Mục tiêu là làm query rõ hơn, rộng hơn hoặc tách thành sub-queries để tăng recall và giảm mismatch giữa query ngắn với tài liệu dài.

1. Query transformation

Biến query ngắn, mơ hồ hoặc nhiều ý thành input dễ retrieve hơn. Đây là lớp query-time sạch nhất, vì Reducto không can thiệp vào đây.

rewrite step-back decomposition
Query gốc ngắn, mơ hồ, hoặc nhiều ý
→
Rewrite / Step-back làm rõ hoặc làm rộng
→
Decomposition tách thành sub-queries
→
Retrieve trên query đã tối ưu

Ba biến thể cốt lõi

  • Query rewriting: viết lại query để cụ thể hơn, thêm ràng buộc, tăng tín hiệu lexical và semantic.
  • Step-back prompting: lùi lên câu hỏi khái quát hơn để lấy bối cảnh nền trước khi drill-down.
  • Sub-query decomposition: bóc tách câu hỏi phức tạp thành nhiều query con để truy hồi toàn diện hơn.

Vì sao hiệu quả

  • Query ngắn thường thiếu ngữ cảnh, còn tài liệu dài lại viết theo phong cách trả lời.
  • Rewriting giảm mismatch giữa ngôn ngữ hỏi và ngôn ngữ tài liệu.
  • Step-back giúp tìm kiến thức nền thay vì chỉ một fact hẹp.
  • Decomposition rất mạnh cho multi-hop và câu hỏi nhiều ràng buộc.
Reducto overlap: không có. Đây là vùng test sạch nhất để đo uplift query-time.
Biến thể Mục tiêu Metric nên nhìn
Rewrite Làm query rõ và giàu tín hiệu hơn. Recall@k, MRR@k.
Step-back Lấy bối cảnh nền trước khi đi vào chi tiết. Context recall, answer coverage.
Decomposition Giải câu hỏi đa ý bằng nhiều query con. Tail query success, answer completeness.

2. HyDE

Hypothetical Document Embeddings: tạo một tài liệu giả định từ query rồi dùng nó để truy hồi. Không đụng Reducto vì đây hoàn toàn là query-time logic.

hypothetical doc dense retrieval zero-shot
Query của user
→
LLM viết hypothetical answer / doc
→
Embed văn bản giả định
→
Retrieve docs gần với hypothetical doc

Ý tưởng cốt lõi

  • Query của người dùng thường là ngôn ngữ hỏi, còn corpus thường là ngôn ngữ trả lời.
  • HyDE dùng LLM để sinh ra một “câu trả lời giả định” hoặc “tài liệu giả định” cho query đó.
  • Embedding của tài liệu giả định thường nằm gần vùng ngữ nghĩa của các tài liệu thật có câu trả lời tương ứng.
  • Chi tiết sai trong bản giả định thường bị encoder làm mờ đi, còn ý cốt lõi thì vẫn được giữ lại.
Vì sao hiệu quả: nó giảm semantic gap giữa query ngắn và tài liệu dài, answer-like.

Khi nào nên dùng

  • Khi dense retrieval theo query literal kém vì cách người dùng hỏi khác cách tài liệu viết.
  • Khi query rất ngắn nhưng muốn tìm đúng phần giải thích trong corpus dài.
  • Khi cần zero-shot và không muốn train retriever riêng.
  • Không lý tưởng cho truy vấn cần exact symbol, math notation, hoặc fact cực kỳ cụ thể nếu LLM dễ hallucinate.
Reducto overlap: không có. Nhưng runtime cost tăng vì phải gọi LLM ở query-time.
Ưu điểm Nhược điểm Đo gì
Zero-shot, dễ cắm vào pipeline hiện tại. Thêm latency và cost cho mỗi query. Recall@k, MRR@k, latency p95, answer faithfulness.
Tốt cho semantic gap và query ngắn. Phụ thuộc vào chất lượng viết giả định của LLM. Hard-query uplift, top-k diversity.

3. HyPE

Hypothetical Prompt Embeddings: chuyển truy xuất thành question-to-question matching bằng cách tạo câu hỏi giả định từ chunk ở giai đoạn indexing.

offline index no runtime latency question matching
Chunk gốc
→
LLM sinh nhiều câu hỏi giả định
→
Embed các câu hỏi đó
→
Retrieve query tới question embeddings

Cơ chế hoạt động

  • Ở offline indexing, mỗi chunk được biến thành nhiều câu hỏi giả định mà chunk đó có thể trả lời.
  • Các câu hỏi này được embed thay vì chỉ embed text gốc.
  • Khi user hỏi, hệ thống embed query và đi tìm các question embeddings gần nhất.
  • Vì cả hai bên đều ở dạng câu hỏi, style mismatch giảm mạnh.
Vì sao khác HyDE: HyDE tạo doc giả định ở runtime, còn HyPE tạo prompts trước ở indexing time.

Khi nào nên dùng

  • Khi corpus khá ổn định và bạn chấp nhận chi phí indexing cao hơn để đổi lấy query-time nhanh.
  • Khi user query thường mang style “hỏi”, còn corpus mang style “trình bày”.
  • Khi muốn cải thiện context precision mà không tăng runtime latency.
  • Không nên bật sớm nếu ingest đã nặng hoặc corpus thay đổi liên tục.
Reducto overlap: không trực tiếp, nhưng Reducto đã làm ingest nặng hơn nên phải cân nhắc chi phí tổng.
Điểm mạnh Điểm yếu Đo gì
Không thêm latency lúc query. Index size và ingest cost tăng. Context precision, recall, retrieval time, index cost.
Tốt cho corpora ổn định, large-scale search. Phụ thuộc prompt quality và coverage của question generation. Offline quality of generated questions, uplift on hard queries.

4. Contextual chunk headers

Gắn ngữ cảnh cấp tài liệu và cấp section vào chunk trước khi embed. Đây là kiểu AutoContext mà dsRAG làm rất rõ, và project của bạn đã có bản rút gọn qua title › section.

document context section path AutoContext-like
Document title / summary / domain
→
Section heading / page range
→
Chunk + header trước khi index
→
BM25 + Dense nhìn thấy context

Nó làm gì

  • Không thay đổi nội dung chunk cốt lõi, mà thêm lớp context ở phía trước.
  • Header thường gồm document title, section title, summary ngắn, domain note hoặc path của section.
  • Khi chunk được embed, vector phản ánh cả nội dung lẫn vị trí logic trong tài liệu.
  • Với sparse search, header cũng giúp BM25 bớt “mù ngữ cảnh”.
Lý do hiệu quả: chunk tự mang theo “ai tôi là” thay vì chỉ có nội dung thô.

Reducto overlap

  • Partial overlap: hệ của bạn đã prepend title › section trước khi index.
  • Partial overlap: Reducto trả về embed_text tối ưu cho embedding.
  • Nếu muốn test clean, nâng cấp header thành summary/section path/domain note và đo uplift riêng.
  • Đây là kỹ thuật rất đáng thử khi chunk đúng nhưng thiếu ngữ cảnh.
Không nên hiểu nhầm: đây không phải semantic chunking. Đây là context injection trước embedding.
Header layer Ví dụ Giá trị
Document-level title, summary, domain, date. Giúp nhận diện chủ đề lớn.
Section-level section heading, page range, subsection path. Giúp chunk đứng đúng vị trí logic.
Local-level neighbor summary, table caption, figure note. Giúp disambiguate chunk rất ngắn.

5. Semantic chunking

Chia tài liệu theo ranh giới ý nghĩa thay vì cắt theo số ký tự cố định. Kỹ thuật này hữu ích cho văn bản dài, nhưng trên PDF path hiện tại nó đã bị Reducto bao phủ một phần.

semantic boundaries sentence embeddings breakpoints
Text → câu / đoạn nhỏ
→
Distance giữa câu / cụm câu
→
Breakpoint theo ngưỡng ngữ nghĩa
→
Chunk coherent theo ý

Nguyên lý hoạt động

  • Không cắt theo độ dài cố định, mà quan sát sự thay đổi ngữ nghĩa giữa các câu hoặc đoạn.
  • Semantic distance tăng mạnh là tín hiệu để đặt breakpoint.
  • Breakpoint có thể xác định bằng percentile, std dev, IQR, heuristics hoặc LLM.
  • Hybrid approach thường thực dụng nhất: dùng embeddings để đo và rule/LLM để xác nhận biên.
Lý do hiệu quả: chunk giữ trọn một ý thay vì bị cắt ngang ở giữa một lập luận hoặc công thức.

Reducto overlap

  • Partial overlap mạnh: Reducto đã chunk theo page_sections, tức là đã xử lý một phần semantic/layout boundary.
  • Vì vậy, test semantic chunking trên PDF path hiện tại sẽ bị trộn với chất lượng parse của Reducto.
  • Để test sạch, hãy dùng text/URL path hoặc baseline fallback không qua Reducto.
  • Trong corpus có heading yếu, semantic chunking thường cải thiện rõ hơn ở long-doc retrieval.
Không nên nhầm: Reducto không thay thế hoàn toàn semantic chunking. Nó chỉ làm tốt một phần boundary detection.
Breakpoint method Ý tưởng Khi hợp
Percentile Cắt tại các điểm có distance nằm trong vùng outlier. Corpus có thay đổi ngữ nghĩa rõ ràng.
Std dev / IQR Dùng thống kê để tìm đoạn có độ lệch lớn. Corpus có cấu trúc vừa phải, cần heuristic ổn định.
LLM / hybrid LLM xác nhận điểm cắt hoặc chỉnh ngưỡng. Corpus phức tạp, cần chất lượng cao hơn tốc độ.

6. Context window enhancement

Sau khi retrieve một chunk, kéo thêm hàng xóm trước/sau nó để phục hồi ngữ cảnh bị cắt. Kỹ thuật này mạnh khi câu trả lời nằm trải trên nhiều chunk liên tiếp.

neighbor chunks local window dedupe overlap
Retrieve chunk top-k
→
Fetch neighbors theo chunk index
→
Merge + dedupe overlap
→
Prompt có context rộng hơn

Ý tưởng cốt lõi

  • Retrieval ban đầu chỉ chọn được chunk “đúng nhất”, nhưng chunk đó có thể thiếu câu trước hoặc câu sau.
  • Window enhancement dùng metadata vị trí để lấy thêm chunk lân cận quanh chunk gốc.
  • Thường cần chunk index, page order, section order hoặc chronological index để xác định hàng xóm.
  • Khi dùng tốt, nó giúp answer đầy đủ hơn mà không cần tăng chunk size quá lớn từ đầu.
Lý do hiệu quả: nhiều câu trả lời cần 1 câu trước để gỡ đại từ, 1 câu sau để chốt kết luận.

Điểm cần chú ý

  • Token budget có thể tăng nhanh nếu window quá rộng.
  • Cần dedupe overlap để không nhét nội dung lặp.
  • Hiệu quả phụ thuộc vào chất lượng chunk boundary ban đầu.
  • Không thay thế retrieval, mà chỉ làm giàu context trước generation.
Reducto overlap: không trực tiếp, nhưng Reducto làm chunk boundary tốt hơn nên window enhancement càng có giá trị.
Window strategy Mục tiêu Trade-off
Fixed neighbors Luôn lấy n chunk trước/sau. Dễ triển khai nhưng có thể over-include noise.
Boundary-aware Dừng theo section/page boundary. An toàn hơn, nhưng có thể thiếu context nếu boundary quá ngắn.
Score-aware Chọn hàng xóm khi chúng đủ liên quan. Thông minh hơn nhưng cần logic tốt hơn.

7. Contextual compression

Retrieve rộng trước, rồi nén lại chỉ còn phần liên quan nhất với query. Mục tiêu là giảm noise và giảm token trước khi vào LLM.

extractor LLM compressor token budget
Retrieve rộng top-k hoặc top-n
→
Compressor giữ span liên quan
→
Prune bỏ câu / đoạn thừa
→
Generate trên context gọn hơn

Nó hoạt động thế nào

  • Không cố đoán chunk nào là tốt nhất ngay từ đầu, mà chấp nhận retrieve rộng.
  • Sau đó một extractor hoặc LLM-based compressor sẽ trích ra câu / span thật sự trả lời câu hỏi.
  • Điều này rất hợp khi tài liệu dài, chunk noisy, hoặc context window bị giới hạn.
  • Compression không thay retrieval, nó là lớp lọc cuối trước generation.
Lý do hiệu quả: mô hình cuối đọc ít hơn nhưng đúng hơn.

Điểm cần cân nhắc

  • Compressor quá aggressive có thể bỏ mất chi tiết quyết định.
  • LLM-based compression thêm cost và latency.
  • Extractor-based compression rẻ hơn nhưng có thể kém linh hoạt hơn.
  • Phù hợp khi bạn đã có retrieval ổn và cần giảm noise ở tầng generate.
Reducto overlap: không trực tiếp. Nhưng vì Reducto đã làm embedding-friendly text, compression càng hữu ích nếu corpus vẫn dài.
Loại compressor Ưu điểm Nhược điểm
Extractor-based Rẻ hơn, predictable hơn. Ít linh hoạt khi query phức tạp.
LLM-based Hiểu câu hỏi tốt hơn, chọn span thông minh hơn. Đắt hơn và có thể sinh noise nếu prompt kém.

8. Relevant Segment Extraction

Không trả về các chunk rời rạc, mà gom các chunk liên tiếp thành một segment có nghĩa. Đây là lớp “answer-ready context assembly” sau retrieval và rerank.

contiguous segment max-sum multi-hop
Top-k chunks sau retrieve/rerank
→
Adjacency scoring + chunk index
→
Segment assembly liên tiếp
→
LLM đọc một đoạn mạch lạc

Lý thuyết cốt lõi

  • Trong tài liệu thật, các mẩu thông tin liên quan thường cluster gần nhau thay vì nằm rời rạc khắp nơi.
  • RSE tận dụng điều đó: thay vì trả về top-k mảnh vụn, nó cố ghép thành một segment liên tục và có ngữ nghĩa.
  • Có thể dùng chunk score, rank, và penalty cho chunk không liên quan để tối ưu hóa chuỗi liên tiếp tốt nhất.
  • Nếu chunking không overlap và có metadata order tốt, việc assembly sẽ đáng tin cậy hơn.
Vì sao hiệu quả: LLM không thích đọc list mảnh; nó thích một đoạn mạch lạc có đầu, giữa, cuối.

Khi nào dùng

  • Multi-hop questions.
  • Questions cần reasoning qua nhiều câu liên tiếp.
  • Long answers cần continuity.
  • Tài liệu kỹ thuật mà một công thức được giải thích qua nhiều đoạn liền nhau.
Reducto overlap: không trực tiếp. Nhưng Reducto làm chunk boundary tốt hơn, nên RSE càng dễ ghép segment đúng.
Signal Ý nghĩa Tác dụng trong RSE
Similarity score Chunk nào gần query hơn. Chọn hạt nhân của segment.
Adjacency Chunk nào đứng cạnh chunk hạt nhân. Giữ continuity cho segment.
Penalty Chunk nào quá nhiễu hoặc không liên quan. Tránh kéo rác vào segment.

Nó làm gì

  • Thay vì chỉ lưu text gốc, hệ thống tạo thêm các câu hỏi mà tài liệu đó có thể trả lời.
  • Các câu hỏi này thường giống style của user query hơn style của văn bản gốc.
  • Kết quả là vector space của document trở nên “dễ chạm” hơn với query dạng hỏi.
  • Đây là một cách mở rộng tài liệu theo hướng retrieval-friendly.

  • Không nên xem đây là bắt buộc nếu dense retrieval hiện tại đã đủ tốt.
  • Phù hợp nhất khi corpus có query sparse hoặc phrasing rất khác văn bản gốc.
Khuyến nghị: đây là technique đáng thử, nhưng phải ablate cẩn thận để tránh đo nhầm lợi ích của Reducto.

10. Hierarchical indices

Tổ chức retrieval theo nhiều tầng: summary -> section -> chunk. Kỹ thuật này thiên về navigation và scaling, phù hợp khi corpus lớn hoặc query cần tìm vùng tài liệu trước rồi mới drill-down.

coarse-to-fine summary layer section drill-down
Summary index tài liệu / chương
→
Section index lọc vùng liên quan
→
Chunk index tìm chi tiết
→
Answer dựa trên vùng đã thu hẹp

Cơ chế

  • Thay vì quét một index phẳng cực lớn, hệ thống tạo một lớp summary hoặc overview trước.
  • Query trước tiên chạy qua lớp coarse để xác định tài liệu hoặc section có khả năng chứa câu trả lời.
  • Sau đó mới search chi tiết trong phạm vi nhỏ hơn ở tầng section hoặc chunk.
  • Coarse-to-fine giúp giảm search space và thường tăng khả năng mở rộng khi corpus lớn lên.
Lý do hiệu quả: không phải mọi query đều cần quét toàn bộ chunk chi tiết ngay từ đầu.

Rủi ro và lưu ý

  • Summary layer có thể drift nếu tóm tắt kém.
  • Update index phức tạp hơn flat retrieval.
  • Cần metadata mapping rõ ràng giữa summary, section và chunk.
  • Không phải kỹ thuật bắt buộc với corpus nhỏ hoặc khi hybrid retrieval hiện tại đã đủ tốt.
Reducto overlap: không trực tiếp, nhưng Reducto làm tốt parse layer; hierarchical indices là lớp retrieval bổ sung ở phía trên.

11. Dartboard retrieval

Chọn top-k không chỉ theo relevance mà còn phải giữ diversity. Mục tiêu là tránh “echo chamber” trong kết quả truy hồi.

relevance diversity greedy selection
Candidate set từ BM25/dense/rerank
→
Greedy pick chunk tốt nhất đầu tiên
→
Penalty cho chunk quá giống nhau
→
Top-k vừa đúng vừa đa dạng

Nó làm gì

  • Retrieval truyền thống thường lấy các chunk rất giống nhau vì similarity cao quanh cùng một đoạn.
  • Dartboard retrieval chọn greedy để mỗi chunk mới vừa liên quan đến query, vừa khác enough so với chunk đã chọn.
  • Có thể xem nó như một lớp diversity-aware selection trên candidate set.
  • Đặc biệt hữu ích khi corpus lặp ý, hoặc khi câu hỏi cần nhiều góc nhìn.
Ý nghĩa: không phải lúc nào chunk có similarity cao nhất cũng là chunk hữu ích nhất để đưa vào prompt.

Khi dùng

  • Khi top-k đang bị gần-duplicate.
  • Khi cần so sánh, phản biện, hoặc tổng hợp nhiều quan điểm.
  • Khi muốn đa dạng hóa evidence trước generation.
  • Không nên dùng nếu corpus đã rất gọn và reranker đã đủ đa dạng.
Reducto overlap: không trực tiếp. Đây là lớp retrieval-selection ở phía trên parse.
Giữ gì Hy sinh gì Đo gì
Relevance nền tảng. Một chút best-match tuyệt đối để đổi lấy diversity. Redundancy ratio, coverage, answer diversity.

12. Adaptive retrieval

Thay vì một chiến lược cho mọi query, hệ thống phân loại query rồi chọn chiến lược phù hợp. Đây là lớp policy, rất hợp với router hiện tại của project.

Ý tưởng cốt lõi

  • Factual: rewrite nhẹ, keyword-heavy, dense + rerank.
  • Analytical: decomposition, RSE, compression.
  • Opinion / compare: dartboard, diversity-aware retrieval.
  • Contextual: window enhancement, session history.
Insight: RAG tốt không phải vì có một kỹ thuật tốt, mà vì biết chọn kỹ thuật đúng cho từng loại query.

Fit với project hiện tại

  • Router hiện tại đã có no_retrieval / simple / complex, nên adaptive retrieval là bước mở rộng tự nhiên.
  • Chỉ cần thêm classification signals và policy rõ ràng là có thể scale logic routing.
  • Phải log reason code để debug misroute.
  • Đây là lớp không đụng Reducto, nên test rất sạch ở query-time.

Offline evaluation: đo cái gì trước khi rollout

Nếu không có ground truth và ablation, rất dễ “thấy hay” nhưng không biết cái gì thật sự giúp.

Retrieval metrics

  • Recall@k, MRR@k, nDCG@k
  • Context precision / context recall
  • Redundancy ratio trong top-k

Generation metrics

  • Faithfulness / groundedness
  • Answer correctness / exactness
  • Citation accuracy (page / section / chunk)

System metrics

  • Latency p50 / p95
  • Token usage / cost per question
  • No-answer rate / refusal rate

Domain-specific metrics

  • Formula correctness
  • Section hit rate
  • Page-level citation correctness
Khuyến nghị: Tạo một gold set nhỏ nhưng đa dạng: factual, analytical, contextual, comparison, multi-hop, và một ít câu hỏi toán / formula nếu corpus của team có PDF kỹ thuật.
Nhóm câu hỏi Đo chất lượng nào Điều gì dễ nhìn nhầm
Factual Recall@k, citation accuracy, answer exactness. Chunk đúng nhưng answer vẫn dài và lan man.
Analytical / multi-hop Faithfulness, context completeness, chain coverage. Answer nghe hợp lý nhưng thiếu evidence liên kết.
Contextual / conversational Session consistency, window usage, router correctness. Trả lời đúng câu hiện tại nhưng quên ngữ cảnh trước đó.

A/B test matrix

Thử theo từng lớp, không bật tất cả một lúc. Control phải là baseline đang chạy ổn nhất.

Variant Thay đổi Giả thuyết Decision metric
A Baseline hiện tại Đây là control để so sánh. Recall@5, MRR@5, faithfulness

Thiết kế test

  • Chạy cùng một dataset và cùng một hidden evaluation set.
  • Giữ embedding model / reranker / generator cố định khi test 1 technique.
  • Log top-k chunks, reason codes, latency và token usage.

Nguyên tắc ra quyết định

  • Không chỉ nhìn average: hãy nhìn tail queries.
  • Nếu uplift nhỏ nhưng cost tăng nhiều, kỹ thuật đó chưa nên rollout.
  • Technique nào tăng faithfulness nhưng giảm latency quá mạnh thì cần cân nhắc.

Rollout và guardrails

Online test chỉ nên làm khi offline score đã đủ tin cậy.

Shadow
Chạy song song với baseline, không ảnh hưởng user. Dùng để debug routing / retrieval / citations.
10%
Canary traffic. So p95 latency, no-answer rate, faithfulness proxy và phản hồi người dùng.
50%
Nếu metrics ổn định, mở rộng traffic. Tối ưu ngưỡng cutoff cho router và compressor.
100%
Chỉ rollout full khi không có regression đáng kể về chất lượng hoặc cost.

Guardrails phải có

  • Latency p95 không vượt ngưỡng team đặt ra.
  • Hallucination / citation mismatch không tăng.
  • No-answer rate không tăng bất thường.
  • Router misroute rate không tăng khi traffic thật vào.

Feature flags nên tách

  • Query enhancer
  • Chunker / context assembler
  • Retriever router
  • Diversity / compression policy
Rollback trigger: chỉ cần một trong các chỉ số sau xấu đi rõ rệt là phải dừng: p95 latency, citation mismatch, no-answer rate, hoặc hard-query recall.
