Backend — Ingestion Pipeline

1. PDF — Math-aware Extraction (PyMuPDF)

  • Font-aware operator remap: ∑, √, ∏, ∫ từ font CMEX10 được ánh xạ lại đúng, không đụng chữ thường.
  • Geometric space reconstruction: Khôi phục dấu cách bị mất từ khoảng cách giữa char boxes.
  • Fraction reconstruction: Dựng lại phân số a/b từ nét vẽ vinculum (đường kẻ ngang).
  • Heading detection: Nhận diện mục numbered (1.2), Roman (I.), step (Bước N:), keyword (Mục lục, Phụ lục...).
  • Boilerplate removal: Lọc header/footer lặp lại trên nhiều trang (threshold ≥ 50% số trang).
  • Clean artifacts: Loại bỏ (cid:NN), watermark tokens, sửa glyph lỗi (vn_text.py).

2. VLM Fallback (gpt-4o-mini)

  • VLM_PARSE=auto (mặc định): chỉ transcribe trang scan/ảnh (empty text) qua vision API.
  • VLM_PARSE=on: luôn dùng VLM (tốn token, chính xác cao nhất).
  • Cache đĩa: SQLite parse_cache.db theo SHA256 ảnh PNG → re-ingest không tốn token lại.

3. URL & Text

  • URL: trafilatura fetch + extract Markdown (giữ bảng, bỏ comment, favor_recall).
  • Plain text: normalize_structure — NFC unicode, collapse spaces, giữ newline cho heading/paragraph.

4. Chunker

  • Section-aware: Không cắt ngang ranh giới mục (mỗi block từ ingestion đã mang section).
  • Atomic formulas/tables: Dòng chứa ký hiệu toán học hoặc bảng ( | ) được giữ nguyên, không cắt giữa chừng.
  • Sentence packing: Tách câu theo dấu câu tiếng Việt (A-Z À-Ỹ), overlap giữa chunk (default 200 ký tự).
  • Smart split: Câu quá dài cắt tại ranh giới logic (dấu phẩy, chấm phẩy, ngoặc).

Backend — Indexing & Storage

BM25 Index

  • Thư viện rank-bm25 (BM25Okapi).
  • Tokenize tiếng Việt bằng underthesea.
  • Index text được prepend doc_title + section path qua _index_text() để tăng match cho context-poor chunks; text hiển thị/citation vẫn giữ nguyên.
  • Lưu bm25.pkl (pickle chunk_ids + corpus_tokens).
  • Lazy rebuild: _dirty flag; rebuild khi search nếu có add/remove.

Vector Index (turbovec)

  • Thuật toán TurboQuant — nén 4-bit (so với float32 là 8×), SIMD.
  • Lưu vector.tvim (IdMapIndex với uint64 chunk_id).
  • Lazy dim: tự suy ra embed_dim từ lần embed đầu tiên.
  • Bit-width tuỳ chỉnh (mặc định 4-bit qua TURBOVEC_BIT_WIDTH).

Embeddings (OpenAI)

  • Model: text-embedding-3-small (default).
  • L2-normalization để cosine similarity = inner product.
  • Cache đĩa: SQLite emb_cache.db lưu vector float32 theo SHA256 key (model + text).
  • Batch: embed_documents batch_size=128; chỉ gọi API cho text chưa có trong cache.

Đồng bộ Index & DB

  • Mỗi lần load, _load_indexes() so sánh chunk_ids trong index_meta.json (version = 3) với DB.
  • Nếu drift (stale files, partial ingest, schema change) → tự rebuild_indexes() từ DB, không cần gọi API vì embedding đã lưu BLOB trong SQLite.
  • delete_document → xoá DB row → rebuild indexes để đảm bảo sync.

SQLite Schema

  • documents: id, title, source, source_type, n_chunks.
  • chunks: id, document_id, chunk_index, text, page, section, embedding (BLOB float32).
  • sessions: id (UUID hex), title, created_at, updated_at.
  • messages: id, session_id, role, content, citations (JSON), trace (JSON).
  • conversation_summaries: id, session_id, summary, summarized_up_to.

Lý do chọn kỹ thuật & Cấu hình hệ thống

Các lựa chọn dựa trên codebase, eval thực tế và tradeoff lý thuyết — không bịa số liệu

🔍 Retrieval — Tại sao BM25Okapi + TurboQuant?

  • BM25Okapi (thay vì BM25L/Plus): Đơn giản, ổn định, phù hợp với corpus nhỏ–vừa (tài liệu thực tập). BM25L chỉ cải thiện khi document dài hơn trung bình đáng kể — không phải trường hợp của chúng ta.
  • Tokenize: underthesea thay vì split từ đơn giản: Tiếng Việt không có dấu cách tự nhiên giữa từ, underthesea phân tách chính xác từ ghép (vd "quy hoạch", "đường đi an toàn") giúp BM25 không miss keyword quan trọng.
  • turbovec (TurboQuant) thay vì FAISS/Annoy: Dung lượng nhỏ (nén 4-bit so với float32 = 8×, file .tvim vài MB), load nhanh, không cần GPU. FAISS tốt cho hàng triệu vectors nhưng overkill cho 1–2 tài liệu và phụ thuộc nvidia-driver khó setup trên máy dev.
  • Embedding: OpenAI text-embedding-3-small: 1536-dim, chất lượng cao cho tiếng Việt, giá rẻ ($0.02/1M tokens), hỗ trợ dimensions nếu cần giảm dim sau này. (Chưa thử local model tiếng Việt — cần A/B test sau.)
  • L2-normalization: Cosine similarity trở thành inner product → turbovec dùng dot-product nhanh, không cần cosine lookup riêng.

🧩 Fusion & Rerank — Tại sao RRF k=60 + bge-reranker-v2-m3?

  • RRF k=60: Theo lý thuyết, k nhỏ phạt mạnh rank thấp, k lớn làm phẳng ranking. k=60 là giá trị phổ biến trong nghiên cứu RRF cho corpus vừa. (Chưa tune k trên dataset riêng — cần thử k=20, 40, 100 trong sprint sau.)
  • Reranker: BAAI/bge-reranker-v2-m3: Cross-encoder đa ngôn ngữ, kích thước model vài GB, chạy local không tốn API token. Eval thực tế cho thấy Hybrid+RRF đã đạt Hit@5=1.000, reranker chỉ cải thiện MRR@5 (0.852 → 0.863) chứ không tăng recall thêm.
  • Top_k tiered: BM25_TOP_K=30, DENSE_TOP_K=30 → RERANK_TOP_N=20 → FINAL_TOP_K=5. Để reranker có đủ candidates (20), nhưng chỉ đưa 5 chunk vào context LLM để tránh noise + tiết kiệm token.

🧠 Memory & Agent — Tại sao summary + window?

  • SUMMARY_THRESHOLD=12: Khi hội thoại > 12 turns, lịch sử dài làm prompt quá lớn, tăng cost và có thể làm LLM "quên" system prompt. Tóm tắt giúp giảm prompt đáng kể so với full history. (Chưa đo chính xác số tokens của summary — cần benchmark.)
  • HISTORY_WINDOW=6: Giữ nguyên 6 tin nhắn gần nhất (3 vòng hỏi-đáp) để agent nhớ ngữ cảnh immediate. (Chưa thử window khác — cần A/B test window=2, 4, 8.)
  • gpt-4o-mini cho sub-tasks: Router, Planner, Distill, Verify, Sufficiency, Replan đều dùng model rẻ. Chỉ Synthesize + Verify Answer dùng model mạnh hơn. (Chưa đo cost chính xác giữa 2 cấu hình — cần benchmark sau.)
  • Replan max 3: Đủ để khắc phục thiếu info (thường 1–2 lần), tránh loop vô hạn. Early stop nếu số bước thất bại không giảm.

📦 Storage & Ingestion — Tại sao SQLite + PyMuPDF?

  • SQLite thay vì PostgreSQL/Chroma: Zero-config, file-based, dễ backup, không cần chạy service riêng. Phù hợp scale hiện tại (tài liệu thực tập, không phải enterprise). Chroma tốt hơn cho metadata filtering phức tạp nhưng thêm dependency nặng.
  • PyMuPDF (fitz): Trích xuất text nhanh, hỗ trợ geometric reconstruction (phục hồi dấu cách, phân số), font-aware remap (∑√∏). pdfplumber không hỗ trợ font remap nên miss công thức toán.
  • Chunk overlap 200 ký tự: Đủ để giữ ngữ cảnh câu bị cắt ngang, không quá lớn gây duplicate. (Ước tính: tiếng Việt trung bình ~40 ký tự/câu → ~5 câu overlap — chưa đo chính xác trên corpus.)
  • VLM fallback (gpt-4o-mini vision): Chỉ dùng khi trang scan/empty text. VLM_PARSE=auto chỉ gọi VLM cho trang cần thiết, tiết kiệm hơn on (luôn gọi). Cache đĩa tránh re-ingest tốn token.

Backend — Retrieval Pipeline (Chi tiết)

Quy trình chính xác trong KnowledgeBase.retrieve():

1. bm25_hits   = self.bm25.search(query, top_k=BM25_TOP_K)          # default 30
2. dense_hits  = self.vector.search(query_vector, top_k=DENSE_TOP_K) # default 30 (nếu ready)
3. fused      = reciprocal_rank_fusion(bm25_hits, dense_hits, k=RRF_K)  # default k=60
4. candidates = fused[:RERANK_TOP_N]                                 # default 20
5. meta       = self.repo.get_chunks([h.chunk_id for h in candidates])
6. IF use_reranker:
      pairs     = [(cid, meta[cid]["text"]) for cid in candidates]
      reranked  = self.reranker.rerank(query, pairs, top_k=len(pairs))
      rerank_map = {cid: sc for cid, sc in reranked}
   ELSE:
      rerank_map = {}
7. final_score = rerank_map.get(cid, h.rrf_score)
8. ordered     = sorted(candidates, key=final_score, reverse=True)[:FINAL_TOP_K] # default 5
9. Trả về list[RetrievedChunk] kèm metadata đầy đủ (doc_title, page, section, text, bm25_score, dense_score, rerank_score)

Reciprocal Rank Fusion

def reciprocal_rank_fusion(bm25, dense, k=60):
    hits = {}
    for rank, (cid, score) in enumerate(bm25, 1):
        hits[cid].rrf_score += 1/(k+rank)
        hits[cid].bm25_score = score
    for rank, (cid, score) in enumerate(dense, 1):
        hits[cid].rrf_score += 1/(k+rank)
        hits[cid].dense_score = score
    return sorted(hits.values(), key=lambda h: h.rrf_score, reverse=True)

Kết hợp xếp hạng từ 2 nguồn độc lập. BM25 bắt keyword chính xác; Dense bắt ngữ nghĩa. RRF không cần normalize score.

Cross-encoder Reranker

  • Model: BAAI/bge-reranker-v2-m3 (kích thước vài GB, lần đầu tải tự động).
  • Chạy local, không tốn token LLM.
  • Lazy load + graceful degrade nếu thiếu weight (return None, fallback rrf_score).
  • Thread-safe: dùng _infer_lock vì HF forward không thread-safe.
  • Tắt được qua USE_RERANKER=false trong .env.

Backend — Controllable Agent Graph

Các node chính

  • Router: Phân loại no_retrieval / simple / complex bằng LLM fast. Nếu chưa có tài liệu → buộc tra cứu.
  • Planner (complex): Tách câu hỏi thành tối đa 4 câu hỏi con độc lập.
  • Retrieve: Gọi kb.retrieve(subq) cho từng câu hỏi con.
  • Distill: Chắt lọc thông tin liên quan từ retrieved chunks cho từng subquestion. Trả KHÔNG_LIÊN_QUAN nếu không liên quan.
  • Verify (Self-RAG): Kiểm tra ghi chú distill có bám nguồn hoàn toàn vào ngữ cảnh không.
  • Sufficiency check: Kiểm tra ghi chú đã kiểm chứng có đủ trả lời câu hỏi gốc không.
  • Replan: Nếu thiếu thông tin và chưa đạt giới hạn lặp, đặt lại câu hỏi con cụ thể hơn.
  • Synthesize: Tổng hợp câu trả lời từ ngữ cảnh đã chắt lọc + ghi chú đã kiểm chứng.
  • Verify Answer: Kiểm tra câu trả lời cuối có hallucinate không. Nếu có → regenerate (max 1 lần).

Xử lý song song

  • Retrieve song song: ThreadPoolExecutor(max_workers=min(4, len(subqs))) chạy kb.retrieve() cho nhiều subquestions cùng lúc.
  • Distill + Verify song song: ThreadPoolExecutor chạy _distill_and_verify cho từng subquestion song song.
  • Giảm latency đáng kể khi câu hỏi phức tạp có nhiều bước.

Replanning Loop

  • MAX_REPLAN_ITERS=3 (configurable).
  • Early stop: nếu số bước thất bại không giảm so với vòng trước → dừng sớm.
  • Không lặp lại nguyên câu hỏi con đã thất bại; phải diễn đạt cách khác.
  • Giữ nguyên các bước đã thành công, chỉ thay thế bước thất bại.

Memory / Conversation Summarization

  • Khi session vượt SUMMARY_THRESHOLD=12 tin nhắn → tóm tắt hội thoại cũ.
  • Giữ lại HISTORY_WINDOW=6 tin nhắn gần nhất nguyên văn.
  • Tóm tắt lưu vào conversation_summaries (SQLite).
  • Agent nhận summary qua system message khi generate answer.

Backend — Prompts & Anti-Hallucination

Prompts trong prompts.py

  • ROUTER_SYSTEM: Phân loại simple/complex/no_retrieval.
  • PLANNER_SYSTEM: Tách câu hỏi phức tạp thành subquestions (tối đa 4).
  • DISTILL_SYSTEM: Chắt lọc thông tin liên quan, giữ số liệu/công thức.
  • VERIFY_SYSTEM: Kiểm tra groundedness (true/false + reason).
  • SUFFICIENCY_SYSTEM: Kiểm tra độ đủ thông tin (sufficient + missing_aspects).
  • REPLANNER_SYSTEM: Đặt lại câu hỏi cho bước thất bại.
  • ANSWER_SYSTEM: Quy tắc trả lời — chỉ dùng ngữ cảnh, trích dẫn [số], LaTeX, chain-of-thought.
  • ANSWER_VERIFY_SYSTEM: Kiểm tra từng khẳng định trong câu trả lời cuối.
  • REGENERATE_SYSTEM: Tái sinh câu trả lời sau khi phát hiện hallucination.
  • SUMMARIZE_SYSTEM: Tóm tắt lịch sử hội thoại.

Lớp chống bịa đặt

  • Prompt ràng buộc: "Chỉ dùng thông tin trong NGỮ CẢNH. Tuyệt đối không dùng kiến thức ngoài tài liệu."
  • Không tìm thấy: Nếu ngữ cảnh thực sự không chứa thông tin → trả đúng nguyên văn "Không tìm thấy thông tin trong tài liệu."
  • Verify groundedness (node verify): Kiểm tra ghi chú chắt lọc trước khi tổng hợp.
  • Verify answer (node verify_answer): Kiểm tra câu trả lời cuối; nếu phát hiện khẳng định không bám nguồn → regenerate (max 1 lần, configurable).
  • Citations: Mỗi ý kèm [số] ánh xạ tới đoạn nguồn. UI hiển thị hover + click drawer.

build_context

def build_context(chunks, label_key="label"):
    parts = []
    for c in chunks:
        loc = []
        if c.get("doc_title"): loc.append(c["doc_title"])
        if c.get("page"): loc.append(f"trang {c['page']}")
        head = f"[{c[label_key]}]" + (f" ({', '.join(loc)})" if loc else "")
        parts.append(f"{head}\n{c['text']}")
    return "\n\n".join(parts)

  • Agent.run() trả về Iterator[dict]; FastAPI bọc trong StreamingResponse (media_type=text/event-stream).

  • Timeout 25s giữa các event; nếu quá thì gửi : keepalive.
  • Lưu assistant message ngay khi nhận event final (survive client disconnect).
  • Fallback persist: nếu disconnect trước khi nhận final, vẫn lưu message ở cuối stream.

  • Reload trang → tự động fetch lại messages từ /api/sessions/{sid}.

Conclusion

Đã đạt được

  • Hệ thống RAG Agent tiếng Việt chạy end-to-end: ingestion → indexing → hybrid retrieval → controllable agent → streaming UI.
  • Math-aware PDF parsing (PyMuPDF) + VLM fallback + section-aware chunking.
  • Hybrid search (BM25 + turbovec + RRF + cross-encoder reranker) đạt Hit@5 = 1.000, MRR@5 = 0.863.
  • Agent graph tất định với replan loop, verify, sufficiency, answer regeneration — chống hallucination đa lớp.
  • UI React hiển thị realtime Agent Graph + Trace, LaTeX, citations hover/click, session persistence.
  • Tối ưu chi phí: adaptive router, model rẻ cho sub-tasks, embedding cache, local reranker.
  • Memory system: conversation summarization khi session dài, giữ nguyên recent window.

Các phần cần làm / cải tiến

  • HyDE: Hiện tại USE_HYDE=false (placeholder trong config). Cần implement HyDE query expansion để cải thiện recall cho câu hỏi mơ hồ.
  • Tinh chỉnh trọng số RRF: Hiện tại RRF dùng k=60 cố định; có thể tune k và trọng số BM25/dense theo loại query.
  • Agent Graph Interactive: Cho phép user click node để xem chi tiết internal state (subquestions, retrieved chunks, distill notes).
  • Citation Highlight trong Drawer: Highlight đoạn text cụ thể được trích dẫn thay vì hiển thị toàn bộ chunk.
  • Multi-document QA nâng cao: Hiện tại retrieve trên toàn bộ KB; cần thêm routing chọn document liên quan trước khi retrieve.
  • Tuning Reranker threshold: Thêm ngưỡng score để loại bỏ candidates yếu, tránh đưa noise vào context.
  • Unit tests: Chưa có test tự động cho agent graph (chỉ có eval retrieval + manual workflow test).
  • Router tuning: Cải thiện regex greeting để tránh false negative (vd "Cảm ơn bạn nhiều" bị ép tra cứu).
  • Deploy: Dockerfile + docker-compose để deploy dễ dàng hơn.

Nhiệm vụ toàn đội tiếp theo — Workflow nghiên cứu & cải tiến

Quy trình chuẩn mà toàn đội sẽ áp dụng cho mọi kỹ thuật RAG mới

🔬 Bước 1 — Nghiên cứu kỹ thuật

  • Đọc kỹ repo RAG_Techniques của Nir Diamant (và các paper/blog liên quan).
  • Tập trung vào các kỹ thuật phù hợp tiếng Việt: HyDE, Query Rewriting, RAP-Fusion, Contextual Compression, Metadata Filtering,...
  • Phân loại: pre-retrieval (query expansion, reformulation) vs retrieval (hybrid, reranking) vs post-retrieval (compression, RAG-Fusion).

📚 Bước 2 — Hiểu sâu lý thuyết

  • Nắm vững mechanism của từng kỹ thuật: input/output, hyperparameters.
  • Phân tích tradeoff: latency vs recall, token cost vs accuracy, compute vs precision.
  • Xác định điểm mạnh / điểm yếu và điều kiện áp dụng (vd HyDE tốt cho query ngắn/mơ hồ nhưng tốn token; reranker tốt cho precision nhưng tăng latency).

🗂️ Bước 3 — Tạo bộ dataset chuẩn

  • Chọn 1–2 file PDF ngắn có cấu trúc rõ ràng (có mục lục, công thức, bảng, câu hỏi cuối chương).
  • Xây dựng ít nhất 3 bộ theo nhãn:
    • Easy: Single-hop, câu hỏi tường minh (vd "Chương 1 đề cập chủ đề gì?", "Trang 3 nói về công thức nào?").
    • Medium: Multi-hop nhưng câu hỏi rõ ràng (vd "So sánh công thức X ở trang 2 và Y ở trang 5").
    • Hard: Multi-hop ngắn gọn, yêu cầu suy luận + expected câu trả lời chất lượng cao; metadata questions (vd "Có bao nhiêu ràng buộc trong hàm mục tiêu ở trang cuối?", "Có bao nhiêu câu hỏi trắc nghiệm ở đầu file?", "Tóm tắt chương 2 trong 3 câu").
  • Mỗi bộ ghi rõ question, expected, difficulty, type (single-hop / multi-hop / metadata).

🧪 Bước 4 — Test toàn diện trên hệ thống hiện tại

  • Chạy toàn bộ dataset qua agent hiện tại (không đổi code) để lấy baseline.
  • Note lại đầy đủ: route, latency, tokens, citations, verify_answer events, regenerated, partial, answer đầy đủ.
  • So sánh answer với expected để đánh giá: đúng/sai/thiếu/bịa.

📊 Bước 5 — Phân tích & Đề xuất cải tiến

  • Thành công ở đâu? Case nào hệ thống xử lý tốt (single-hop, câu có keyword rõ).
  • Fail ở đâu? Case nào trả lời sai/thiếu/hallucinate (multi-hop, metadata, câu mơ hồ).
  • Root cause: Do retrieval (không tìm được chunk)? Do agent (planner/reasoner)? Do prompt?
  • Đề xuất: Chọn 1–2 kỹ thuật từ bước 1 phù hợp nhất để khắc phục root cause.

💻 Bước 6 — Code triển khai + A/B Testing

  • Viết code triển khai kỹ thuật đã chọn (configurable qua .env, không break backward compat).
  • A/B Testing: Chạy lại dataset với cấu hình mới (A) vs baseline cũ (B).
  • So sánh metrics: Recall@5, MRR@5, Hit@5 (retrieval) + accuracy, latency, token cost (end-to-end).
  • Chỉ merge khi có cải thiện đo được (statistically significant hoặc ít nhất trend rõ ràng).

🔒 Bước 7 — Unit Test + Docs + Pull Request
